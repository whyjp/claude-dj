왜 Claude DJ인가

멀티 모니터, 멀티 세션, 하나의 덱

터미널 여러 개를 띄워 놓고 동시 작업을 병행하는 워크플로에서, Claude의 질문에 일일이 대응하는 것은 반복적이고 소모적이다.

01

흐름이 끊기지 않는다

Claude가 물어와도 에디터에서 손을 뗄 필요가 없다. 시선은 코드에 두고, 옆에 놓인 덱 버튼만 누르면 응답 완료. 컨텍스트 스위칭 비용을 제거한다.

02

모든 세션, 한 곳에서

메인 작업, 사이드 브랜치, 별도 프로젝트 — 세션이 몇 개든 응답 대기 중인 세션이 자동으로 덱 위에 올라온다. 화면을 찾아 헤매지 않는다.

03

덱이든 터치든

Ulanzi D200 물리 버튼, 브라우저 가상 덱, 터치 모니터까지 — 환경에 맞는 인터페이스를 고르면 된다. 입력 수단이 바뀌어도 동작은 같다.

확장성

어떤 장비든, 덱이 된다

Claude DJ의 클라이언트는 WebSocket 하나로 Bridge에 연결된다. 입력 장치의 형태에 제약이 없다.

A

물리 덱

Ulanzi D200 같은 USB 스트림 덱. 물리 버튼의 촉각 피드백으로 시선 없이도 조작할 수 있다. UlanziDeck SDK를 통해 Bridge와 통신.

B

브라우저 가상 덱

별도 장비 없이 브라우저에서 Virtual DJ를 띄운다. 가장 빠르게 시작할 수 있는 방식. 로컬 Bridge에 WebSocket으로 직결.

C

터치 모니터 직결

작업 PC에 터치 모니터를 연결하고, 웹앱을 풀스크린으로 띄운다. 넓은 화면에서 선택지를 한눈에 확인하면서 터치로 바로 응답.

D

리모트 디바이스

자체 CPU를 가진 Android 태블릿이나 미니 PC가 네트워크를 통해 Bridge와 통신한다. 작업 PC와 물리적으로 분리된 독립 덱 단말로 작동.

동작 원리

다섯 단계로 완성되는 루프

00

스킬 주입

choice-format 스킬이 세션에 자동으로 주입된다. 이후 Claude는 모든 결정을 AskUserQuestion 도구 호출로 출력한다.

01

Hook 이벤트 수신

AskUserQuestion 호출이 PermissionRequest 훅을 발화하면, Bridge가 세션을 WAITING_CHOICE로 전환한다. 일반 도구라면 WAITING_BINARY. 응답까지 최대 60초 블로킹.

02

레이아웃 전환

상태 변경 즉시 WebSocket으로 새 레이아웃을 브로드캐스트한다. 크로스세션 포커스 필터링 덕분에, 다른 세션의 작업이 현재 선택 화면을 덮어쓰지 않는다.

03

버튼 입력

덱에서 선택지를 확인하고 버튼을 누른다. 포커스 이동도, 타이핑도 없다. 응답은 포커스된 세션에 정확히 라우팅된다.

04

응답 전달

Bridge가 훅 응답으로 선택 결과를 반환하고, Claude Code는 곧바로 해당 선택지에 따라 다음 단계를 진행한다.

핵심 메커니즘

스킬 하나로 Claude의 출력 형태를 바꾸다

Claude DJ의 핵심은 choice-format 스킬이다. 이 스킬 하나가 Claude의 행동을 모델 수준에서 바꾼다 — 모든 결정 지점에서 구조화된 AskUserQuestion 도구를 호출하게 만드는 것.

기존 (스킬 없음)

텍스트 기반 선택

Claude가 번호 목록을 텍스트로 출력한다. 사용자는 터미널에서 직접 숫자를 타이핑해야 하고, 덱은 이를 감지할 수 없다.

Which approach? 1. Refactor 2. Rewrite 3. Patch → 터미널 포커스 필요
스킬 주입 후

AskUserQuestion 도구 호출

Claude가 AskUserQuestion 도구를 호출한다. 이것이 PermissionRequest 훅을 트리거하며, 유일하게 블로킹되는 훅이다.

AskUserQuestion({ options: [ "Refactor", "Rewrite", "Patch" ] }) → 덱 버튼 자동 표시

핵심 구분

선택과 확인은 다르다

스킬은 Claude가 계획 설명을 선택지처럼 나열하는 안티패턴을 원천 차단한다. "이대로 할까요?"는 선택이 아니라 확인이다.

진짜 선택 (Real Choice)

서로 다른 경로

2~4개의 진정한 대안이 존재하고, 어떤 옵션을 고르느냐에 따라 결과가 달라진다.

[1] Refactor [2] Rewrite [3] Patch → 3개 버튼, 각각 다른 결과
확인 (Confirmation)

계획 승인 요청

계획을 텍스트로 설명한 뒤, "진행 / 다른 방향" 두 가지로 승인을 묻는다.

계획: Pioneer에 상위 모델 적용 [1] 진행 [2] 다른 방향 → Y/N 바이너리, 간결
안티패턴 (금지)

계획을 선택지로 위장

계획 설명과 추가 범위를 같은 레벨의 선택지로 섞어 놓는 형태. 사용자가 의미 있게 "선택"할 수 없다.

✗ "X를 수정해서 적용" ✗ "기존 scaffold에도 적용" → 범위 번들링. 계획을 먼저 설명, 그 다음 Y/N으로 물어야 함

상태 머신

상태가 곧 레이아웃이다

세션 상태가 전환되는 순간, 덱의 버튼 배치가 통째로 바뀐다. 작업 중에는 버튼이 웨이브로 흘러, 지금 무엇이 돌고 있는지를 시각으로 보여준다.

IDLE

대기

작업 완료 또는 입력 대기 상태. 모든 버튼이 비활성화된다.

PROCESSING

작업 중

도구 실행이 감지되면 버튼들이 시차를 두고 점멸하며, 진행 중임을 시각적으로 보여준다.

WAITING_BINARY

권한 승인 대기

권한 요청 수신. Approve와 Deny만 활성화되고, 나머지 버튼은 비활성.

WebSocket Protocol

Bridge ↔ 클라이언트 통신 규약

Bridge Server와 클라이언트(Virtual DJ 또는 Ulanzi D200) 간 주고받는 WebSocket 메시지 스키마.

← Bridge → Client: LAYOUT (전체 상태)
{ "type": "LAYOUT", "preset": "binary" | "choice" | "processing" | "idle" | "response", "sessionCount": 3, "session": { "id": "abc", "name": "api-server", "state": "WAITING_CHOICE" }, "agent": { // 서브에이전트 (null=root) "agentId": "ag1", "type": "Explore", "state": "PROCESSING" }, "agentCount": 2, // 현재 세션의 자식 수 "focusSwitched": true, // slot 11/12 전환 시 "prompt": { ... }, // binary preset "choices": [ ... ] // choice/response preset }
← Bridge → Client: 기타 메시지
// 전체 dim (세션 종료 시) { "type": "ALL_DIM" } // 초기 핸드셰이크 { "type": "WELCOME", "version": "0.1.0", "sessions": [ { "id": "abc", "name": "api-server", "state": "PROCESSING", "agents": [ { "agentId": "ag1", "type": "Explore", "state": "PROCESSING" } ] } ] }
→ Client → Bridge: 입력
// 버튼 프레스 { "type": "BUTTON_PRESS", "slot": 0, "timestamp": 1743000000000 } // 클라이언트 준비 { "type": "CLIENT_READY", "clientType": "virtual", "version": "0.1.0" } // 슬롯 매핑: // slot 0 = Approve (binary) 또는 Choice 1 // slot 1 = Deny/Always 또는 Choice 2 // slot 11 = 세션 전환 (cycleFocus) // slot 12 = 에이전트 전환 (cycleAgent)
← Hook 응답 (permission.js stdout)
// AskUserQuestion → 선택 번호 반환 { "hookSpecificOutput": { "hookEventName": "PermissionRequest", "decision": { "behavior": "allow", "updatedInput": { "answer": "2" } } } } // 일반 Permission → allow/deny/alwaysAllow { "hookSpecificOutput": { "hookEventName": "PermissionRequest", "decision": { "behavior": "allow", "message": "Claude DJ: allow via button" } } }
